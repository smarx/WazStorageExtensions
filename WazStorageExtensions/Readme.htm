Transient Fault Handling Framework for SQL Azure, Windows Azure Storage,  Service Bus, Caching Service

Version 1.8                
October 2011

The Transient Fault Handling Framework solution provides a reusable framework for building extensible retry policies capable of handling different types of transient conditions in applications leveraging SQL Azure, Windows Azure storage (queues, blobs, tables), Windows Azure Service Bus and Windows Azure Caching Service.

The  main capabilities provided by the framework include the following:

  • Provides the foundation for building highly extensible retry logic for handling a variety of transient conditions, not limited to SQL Azure.

  • Supports a range of pre-defined retry policies (fixed retry interval, progressive retry interval, random exponential backoff).

  • Supports separate retry policies for SQL connections and SQL commands for additional flexibility.

  • Supports retry callbacks to notify the user code whenever a retry condition is encountered.

  • Supports the fast retry mode whereby the very first retry attempt will be made immediately thus not imposing delays when recovering from short-lived transient faults.

  • Enables to define the retry policies in the application configuration files.

  • Provides extension methods to support retry capabilities directly in SqlConnection and SqlCommand objects.

Release History

The following list highlights all major updates and improvements that were introduced in the framework since its original release date:

  • Version 1.0 (June 2010):

    • Initial version supporting transient conditions for SQL Azure only.

  • Version 1.1 (October 2010):

    • Added support for handling transient conditions in Windows Azure queues, blob containers, Windows Azure Service Bus (including SB messaging and Durable Message Buffers).

    • Introduced configuration classes to be able to configure the retry policies in the application configuration for added flexibility.

    • Added support for Fast Retry mode whereby the first retry attempt is not subject to delay.

  • Version 1.2 (December 2010):

    • Enhanced SqlAzureTransientErrorDetectionStrategy with additional transient conditions. For the purposes of increased reliability, the framework assumes that DBNETLIB error code 20 (Encryption Not Supported) is transient.

  • Version 1.3 (January 2011):

    • Improved configuration parameter validation to ensure that invalid values such as negative delay interval will not cause reliability issues in the framework.

    • Removed error codes 40549, 40550, 40551, 40552 and 40553 as per suggestion from the test team. These errors are unlikely to be subject to recovery through a generic retry.

  • Version 1.4 (March 2011):

    • Included a documentation file in CHM format.

    • Added default policies for all 3 supported retry methods. The default retry policy treats all caught exceptions as transient errors.

  • Version 1.5 (June 2011):

    • Added a component for decoding SQL Azure engine throttling reason codes.

    • Added a new implementation of ITransientErrorDetectionStrategy that can recognize transient faults when working with Windows Azure Caching Service.

  • Version 1.6 (July 2011):

    • Added support for retry logic in asynchronous operations. This enables handling transient faults when invoking asynchronous APIs.

  • Version 1.7 (September 2011):

    • Updated ServiceBusTransientErrorDetectionStrategy with additional transient faults that are specific to the latest release of the Services Bus.

  • Version 1.8 (October 2011):

    • Updated ServiceBusTransientErrorDetectionStrategy with a new transient fault.
